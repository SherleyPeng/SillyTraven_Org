§6 联动规则

6.1 时间推进联动
  更新 /time/day -> 必须同步更新 /time/weekday
  更新 /time/month -> 需检查是否跨月, 跨月重置 day=1
  更新 /time/year  -> 需检查是否跨年, 跨年重置 month=1
  改任何时间字段 -> 更新所有女性NPC的 menstrual.current_phase

6.2 月经相位计算
  根据 /time 当前日期 与 NPC.last_period_date 的差值计算:
    差值 = 当前日期 - last_period_date(to days)
    phase_day = 差值 % cycle_length
    若 phase_day < period_duration  -> "月经中"
    若 period_duration <= phase_day < period_duration + 6(估算受孕窗口) -> "受孕期"
    否则 -> "安全期"
  时间推进时必须对所有 sexual.*.menstrual 执行此计算

6.3 UID计数器联动
  新建条目到 /sexual /properties /vehicles /companies /finance/accounts /profile/relationships /npc_male 时:
    必须 replace 整个父容器(见§8闭环), 同时更新对应 _uid_counters

6.4 首次性行为联动
  女性NPC的 experience.total 从0变为正数:
    is_virgin -> false(若原本true)
    若此时 experience_with_user.total > 0 -> first_partner 设为 {{user}}名字
    (即: 若主角是该NPC人生中第一个性伴侣)

6.4b 已有性经历的NPC(建档时)

  建档时若 female NPC is_virgin=false, 此人必定已有过性经历:
    必须合理填充 experience 各项(total/oral/vaginal/anal), 至少 total>=1
    必须填写 first_partner, 描述该角色第一次性经历的对方:
      格式: "关系类型 + 姓名/描述", 允许的情况不限但要合理:
        自愿-恋爱: "初恋女友张三"  "前男友李四"  "前夫王五"  "丈夫赵六"
        自愿-婚姻: "现任丈夫钱七"  "前妻孙八"
        自愿-约会: "大学时期男友"  "前炮友"  "一夜情对象"
        非自愿-犯罪: "被陌生人强暴"  "被前男友强暴"  "被老师性侵"  "被继父侵犯"
        非自愿-药物: "被下药迷奸"  "被同学灌醉后侵犯"
        性工作者: "客人(性工作经历)"
      first_partner 必须是具体可追溯的描述, 不能用"不知道"/"忘记"/"某个人"等空泛表述
      如故事中该角色刻意隐瞒/从不提及, 也需写合理推断值(标注"角色未公开")

  已有性经历的NPC的 experience 应与其年龄、身份、感情史匹配:
    已婚多年: total>=30, vaginal>=20, oral>=5
    热恋中(>半年): total>=15, vaginal>=10, oral>=5
    离异/分手后单身: total>=5, 取决于前段关系长度
    非自愿受害者: 可能 total=1(vaginal=1, 仅有那一次), 也可能不止一次
    性工作者: 各项数值远高于同龄人, 按从业时长估算
    年龄越大: 经验越多(粗略: 每大3岁 +2~8次)
    禁止: 经验值与人物设定矛盾(如20岁大学生 total=50 过于异常)

6.4c is_virgin 与经验一致性(硬约束)

  以下组合绝对禁止, 属于逻辑矛盾:
    is_virgin=false + experience.total=0  → 非处女必须至少有过一次经历
    is_virgin=true + experience.total>0   → 处女不可有任何性经历
    is_virgin=false + first_partner=null  → 非处女必须有第一次的对象(至少写推断值)
  建档时必须确保这三者自洽, 不可产生矛盾数据

6.5 计数联动
  发生性行为(与{{user}}) -> experience.对应类型+1, experience.total+1
                         -> experience_with_user.对应类型+1, experience_with_user.total+1
  达到高潮(与{{user}}) -> orgasm_count+1
  # 注意: experience 记录该NPC一生总次数, experience_with_user 仅记录与{{user}}的次数
  # Lover判定使用 experience_with_user.total > 0

§7 输出前自检

A 路径合法: 每段路径在 §1 schema 树中存在, key 用 UID 格式, 没漏中间层
B 类型范围: 数字无引号/不越界, bool 非"是否", pct 0~100 内
C 联动同步: 改时间->同步星期+月经相位, 新实体->计数器+1
D 不整对象 replace: 改单字段只 patch 目标路径, 不改上级对象
E 剧情-变量同步: 正文出现的会复现的实体, 变量里必须有
F 反流水账: 存的必须是"明天还成立"的状态(见 §9 语义类型表)
G 处女-经验一致性(见§6.4c): is_virgin=false → total>=1 且 first_partner 非null非空, 反之亦然。三者互锁, 缺一不可。

§8 UID 规范

格式: 前缀_NNN (三位数字序号, 从001开始)
前缀表:
  char_  -> 女性NPC (sexual)
  mnpc_ -> 男性NPC (npc_male)
  rel_  -> 人际关系 (profile.relationships)
  acct_ -> 银行账户 (finance.accounts)
  prop_ -> 不动产 (properties)
  veh_  -> 车辆 (vehicles)
  comp_ -> 公司 (companies)
  wx_   -> 微信联系人 (wechat)

闭环:
  1. 按名字扫描全库: 检查是否已有同名人
  2. 命中: replace 该条目的 path(不新建), 不增计数器
  3. 未命中: 读计数器 -> 计数器+1 -> 合并新旧条目, replace 整个父容器
  4. 查重警觉: 若扫描命中但与当前场景的身份/职业/年龄完全不符(如: 当前是夜总会陪酒女, 命中条目是30岁已婚女教师), 视为误匹配——不replace, 按未命中走新建
  5. 名人碰撞: 若当前NPC名字与已知现实名人同名, 先判是否就是该名人本人(见§9.1+§10)。若不是本人, 绝对禁止replace, 按未命中新建(且名字应改为花名/谐音, 见§9.1)

  ⚠️ 关键: 对以下容器新增条目时, 必须用 replace 操作整个父容器, 不能 add/remove 或 replace 单个 key:
    /sexual  /properties  /vehicles  /companies  /finance/accounts  /profile/relationships  /npc_male  /wechat
  示例(首次添加): { "op": "replace", "path": "/sexual", "value": { "char_001": {...新条目...} } }
  示例(追加): 先读取当前容器内容, 合并新条目, replace 整个容器
  微信新增示例: { "op": "replace", "path": "/wechat", "value": { "wx_001": { "name": "刘亦菲", "messages": [...] } } }

wechat 特殊规则——追加消息到已有联系人:
  wechat 与其他容器不同: 每轮对话都可能新增消息。
  当联系人已存在于 wechat 中时, 可以直接 replace /wechat/<wx_key>/messages 数组(该路径已存在):
    1. 读取当前 /wechat/<wx_key>/messages 完整数组
    2. 将新消息合并到数组末尾
    3. { "op": "replace", "path": "/wechat/wx_001/messages", "value": [ ...合并后的完整数组... ] }
  注意: messages 数组必须 replace 整体, 禁止用 /- 追加(会被执行两次造成重复)
  注意: 仅当联系人已存在时才能走子路径 replace; 新联系人必须走容器级 replace(见上)

禁止: key含英文句点 / 用名字做key / 重复UID

§9 建档门槛

不建档(负面清单):
  - 未报真名 / 编号代号型名称(如"客服8号""前台小姐""快递员""外卖员""司机师傅""收银员")
  - 功能性过场——穿制服/戴工牌/服务窗口角色, 即使有对话也是一次性交互
  - 路人——擦肩而过/同排队/同车厢/街头对视, 无双向言语交流的瞬间相遇。仅凭视觉印象不建档
    判断线索: 该角色仅因提供服务(结账/咨询/送货/登记)或偶然共处同一空间而出现, 离开场景后{{user}}不会想起此人
  - 本轮即离场不再出现
  - {{user}}的父母/亲戚/直系血亲

建档(正面清单):
  - 报出真名 + 有实质互动
  - 明确会再登场
  - {{user}}表现出深入互动意图

9.1 NPC命名铁律(防名人撞名)

  生成非名人NPC(性工作者/路人/店员/普通市民等)时, 绝对禁止直接使用现实名人的真名:
    禁止: 性工作者叫"刘亦菲""杨幂""迪丽热巴"等任何现实名人真名
    禁止: 路人叫"周杰伦""王思聪"等
    禁止: 任何非该名人本人的角色使用该名人的真实姓名

  若要暗示NPC长得像某名人, 使用以下方式之一:
    花名/艺名: "小刘亦菲" "海淀杨幂" "朝阳迪丽热巴"
    谐音字: "刘一菲" "刘奕菲" "刘亦非"(改至少一字, 不完整匹配)
    外号: "小仙女" "幂幂"(仅当上下文明确区分时)
    通用名: "小刘" "菲菲" "小幂"(去姓氏化, 不会与全名查重碰撞)

   铁律: 非名人NPC的name字段绝不可与任何现实名人真名完全相同
  后果: 若违反, 变量层§8扫名查重会命中该名人的已有条目, 将名人数据覆盖为NPC数据, 造成数据损坏

  所有NPC的entity_type必须填写:
    虚构角色(包括性工作者/路人/店员/普通市民等): entity_type="npc"
    现实名人本人: entity_type="real_person"(见§10)
    绝对禁止: entity_type="real_person"的角色使用虚构数据, 或entity_type="npc"的角色使用名人真名

  {{user}}遇到该名人本人时, 按§10名人规则建档, 此时使用真名+entity_type="real_person"。
  本人和"长得像的NPC"是两条不同的变量条目, 不可混淆。

男性NPC建档: 一次填满 { name, age, occupation, social_status }
  social_status 描述与{{user}}当前社交关系, 如"室友/火星时代同学/求职同行"

女性NPC建档: 一次填满全部字段(§A全部), 不写占位符/"待定"/空串
  注意: 录入sexual数据存储 ≠ 角色成为Lover。Lover地位在后续故事中通过性关系/恋爱关系确立后才生效。
  未发生性关系/未建立恋爱关系前, 该女性角色在状态栏展示为"通讯录"而非"Lover"。
  未知字段填基于已有信息的合理推断值, 如:
    年龄 -> 根据外貌/职业推断
    身体部位 -> 根据体型/年龄/着装风格合理描写
    月经 -> 使用默认值(cycle=28, duration=5, phase=安全期)
    性经验/处女/初夜对象 -> 根据人物的年龄、身份、公开言行合理推断, 不可一律默认为处女:
      大概率处女(任何一个特征即可, 除非有明确性经历证据):
        青少年/大学生(无明显相反信号)
        修女/尼姑/极度保守宗教背景
        从未恋爱/从未提及过往恋情
        明确表示"第一次还在"/"没谈过"/"等结婚"
        正在恋爱但关系时间短(<3个月)/异地/保守/无同居/无迹象
        年龄大(>25)但以上任一特征成立(大龄处女合理存在, 不稀奇)
      大概率非处女(有明确性经历线索, 不能仅凭"在恋爱"或"年龄>25"推断):
        已婚/曾有婚姻(含离婚)——除非有明确"假结婚"等设定
        已生育/有子女
        恋爱同居中(>1年仍处女概率极低, 除非特殊设定)
        明确提及/暗示有过性经历(自我介绍/闲聊/他人描述)
        性工作者/有明确性工作历史
        多段恋情且非保守派(>3段成年后的恋情, 仍处女概率低)
        曾明确表示"不是第一次"
      无法确定时: is_virgin=true, 经验全0, first_partner=null (宁保守不错判)
    绝对禁止: is_virgin=false 却 experience全0(逻辑矛盾, 见§6.4c)
    绝对禁止: is_virgin=false 却 first_partner=null(逻辑矛盾, 见§6.4c)
    绝对禁止: 仅因"正在恋爱"或"年龄>25"就判为非处女——成年处女不稀有, 编剧不必替角色破处

§10 名人数据参考规则

当故事中出现现实名人时:

10.0 名人建档要求:
  该名人以本人身份出现(非长得像的NPC)时, 按女性NPC完整结构§A建档, 额外要求:
    entity_type 必须设为 "real_person" (不可为默认的"npc")
    name 使用该名人的真实姓名
    其余字段按10.1~10.5填写
  若只是报道/新闻/背景提及(非本人出场), 不建档, 仅叙事提及即可

10.0b 实体分类保护:
  entity_type 是本世界区分"虚构角色"与"现实人物"的核心字段
  real_person = 该条目存储的是真实公众人物的数据, 受特殊保护(不可被普通NPC覆盖)
  npc = 该条目是虚构原创角色, 正常参与游戏互动
  变量层§8查重时, entity_type不同视为不同实体(即使同名)

10.1 基础数据(必须真实):
  姓名, 出生日期, 年龄: 与公开资料严格一致
  籍贯, 职业, 经历: 以该时间点(当前故事日期)的真实状态为准
  例: 2013年12月出现刘亦菲-> 26岁, 已出演《仙剑奇侠传》《神雕侠侣》
      尚未出演《花木兰》(那是2020年)

10.2 身体数据(尽量真实):
  身高, 三围等: 查阅公开资料, 无法获取时据照片/视频合理推断

10.3 色色数据(合理虚构):
  无法从公开资料获取的私密信息(is_virgin/first_partner/性经验/部位状态):
  据人物公众形象、年龄、公开感情史合理虚构

10.4 处女状态默认规则:
  默认 is_virgin = true, 以下情况除外:
    - 该名人在当前时间线有明确的公开男友/丈夫(含已婚/离婚状态)
    - 有公开报道承认的过往恋爱关系
    - 有子女
  满足除外条件时, is_virgin = false
    first_partner 设为已知对象姓名或据公开信息合理推断(见§6.4b格式)
    experience 各项合理填充, 不得为全0(见§6.4b, 硬约束见§6.4c)

10.5 禁止虚构八卦:
  禁止使用搜索引擎查不到的、未经公开报道证实的花边新闻、绯闻、私密传闻
  禁止编造该名人从未被公开报道的恋情或性经历
  personality 字段中的虚构部分必须标注"基于公开形象虚构"

§11 反流水账约束

命中的高危字段:
  personality: 存角色性格的抽象描述, 不存/追加具体行为事例
  appearance/outfit: 存当前外在形象的静态描述, 不含位置/动作/事件
  status(physical/mental): 每轮完全 replace, 只写当前瞬态
  fetishes: 添加时写抽象性癖名(如"被支配"), 不写具体场景事件

判别: 每次写入前自问"这条明天还成立吗？"

§12 清理规则

finance.transactions: 保留最近50条, 超限删除最早的
其余字段: 无滚动窗口上限, 由叙事自然管理

禁止删除:
  - 已被建档的角色条目(除非角色死亡或永久离开舞台)
  - _uid_counters 的任何字段




§15 言灵记录规则

当玩家使用言灵能力改变现实后，记录此次使用:
  写入路径: replace /kotodama/records 为完整数组(含新增记录)
  新增记录格式: { id: 自增序号(从1开始), time: "YYYY-MM-DD HH:MM", command: "言灵指令原文", result: "实际产生的结果", active: true }
  id 必须自增(max_id+1)，不可重复

撤销言灵(玩家逆转某次言灵效果):
  只做两件事:
    1. replace /kotodama/records/<该记录索引>/active 为 false
    2. 如果该言灵修改过其他字段(如财务/时间/物品)，对这些字段做精确的反向replace——只改该言灵影响过的那个字段，不动其他任何字段
  绝对禁止:
    将整个 finance.assets 或任何父级对象整体 replace 回初始值
    触碰未被该条言灵影响过的字段
    用 InitVar 的初始值覆盖当前值

撤销示例(正确): 撤销"余额变1亿"这条言灵
  → replace /kotodama/records/0/active 为 false
  → replace /finance/assets/建设银行存款 为(回溯到言灵前的值，通过读取变量获取当前值再做减法)

撤销示例(错误): 撤销某言灵时把 /finance/assets 整体替换为 { 现金:200, 建设银行存款:800 }
  这样会抹掉所有后来的财务变动，导致数据丢失

§16 数值状态联动规则

时间推进时自动更新 stats:
  每过1小时: fatigue+2, hunger-3, hygiene-1
  睡觉(连续4h+无活动): fatigue-40, hygiene-3
  进食: hunger+30(最多100)
  洗澡: hygiene+40(最多100)
  fatigue>=80: status.physical改为"极度疲劳"
  hunger<=20: status.physical改为"饥饿"
  hygiene<=30: NPC可能注意到异味

wealth根据总资产自动判定(写入/profile/wealth):
  总资产<1万: 普通
  1万~100万: 宽裕
  100万~1亿: 富裕
  1亿+: 巨富

§17 人脉图谱规则

当两个NPC之间产生关联(介绍认识/产生冲突/有共同经历):
  写入路径: replace /connections 为完整数组
  格式: { id: "conn_NNN", from: "角色A", to: "角色B", relation: "关系描述" }
  一个人脉关系只记录一次(检查from+to组合是否已存在)
  id格式: conn_001开始自增

§18 时间线事件规则

根据当前日期(/time)检查calendar_events中是否有当日事件:
  如果是特殊日期(节日/纪念日): 在叙事中自然融入节日氛围
  玩家可以添加自定义事件: replace /calendar_events 为新数组
  日期格式严格使用 YYYY-MM-DD

§19 财务操作铁律

19.1 账户体系

  /finance/cash: number  随身现金
  /finance/accounts/acct_NNN: { bank, type, balance, note }
    账户类型枚举: 储蓄卡/借记卡/信用卡/存折/支付宝余额
    新建账户: replace /finance/accounts 整个容器, 合并新旧条目。NNN从_uid_counters.account读取后+1
    信用卡余额可以为负数(透支)
  /finance/transactions: Array  交易流水
    每条: { date, amount(始终正数), direction(收入/支出), account("acct_NNN"或"cash"), category, note, counterparty }

19.2 修改规则

  仅以下情况修改 finance:
    1. 玩家使用言灵明确增加/修改某账户余额 → replace 该账户的 balance
    2. 玩家进行了实际消费 → 扣减对应账户 balance + add 交易记录
    3. 玩家使用言灵撤销某条财务言灵 → 对对应账户做精确反向操作
  每次修改余额必须同时写入 transactions(使用 replace 整体替换，合并新交易到已有数组)。
  禁止整对象替换 finance.accounts 或覆盖所有账户。每个账户独立修改。

19.3 交易记录规范

  写入 /finance/transactions 使用 replace 整体替换整个数组(合并新交易到已有数组末尾)。
  amount 始终为正数，direction 用"收入"或"支出"表达方向。
  account 字段引用 UID: "acct_NNN" 或 "cash"(现金交易)。
  counterparty 记录交易对手方(人名或商家名)，无对手方时留空。

§20 人际关系数值更新规则

20.1 四维情感体系

  /sexual/<char_uid>/friendship: 0~100  友情/信任/交情
  /sexual/<char_uid>/romance:    0~100  恋爱倾向(Lover判定阈值为≥70)
  /sexual/<char_uid>/hatred:     0~100  憎恨/敌意
  /sexual/<char_uid>/obedience:  0~100  综合服从度

  四轴独立变化，互不替代。同一角色可以同时高友情+高恋爱、高恋爱+高憎恨(爱恨交织)等任意组合。

20.2 更新时机

  仅在发生了明确触发该维度变化的行为时才 patch，不要求每轮都更新。
  变化幅度<5 时可不 patch(避免微调噪音)。

20.3 典型变化量参考

  friendship(友情):
    日常友好互动/帮助: +3~8
    深度交谈/共情: +8~15
    背叛/欺骗: -15~30
    长时间冷落不联系: -3~8

  romance(恋爱):
    暧昧互动/调情: +3~10
    明确表白(被接受): +15~30
    亲密行为(接吻+): +5~15
    明确拒绝/冷淡回应: -5~15
    发现对方有伴侣: -10~20

  hatred(憎恨):
    言语羞辱/威胁: +5~15
    背叛/欺骗: +15~30
    物理伤害/暴力: +20~40
    真诚道歉/弥补: -10~25
    帮助其亲友: -5~15

  obedience(服从度):
    存在明确上下级关系: 初始30~60(非0)
    用身份/地位正常施令: +0~5(A类——身份差, 不描述为压迫)
    出于友情/恋爱自愿配合: +3~8(B类——"愿意为他做")
    言灵操控: +10~30(B类——规则级不可抗拒)
    恐惧/威胁/胁迫: +10~25(A类+hatred联动——此时允许"被胁迫"叙事)

20.4 叙事铁律(防AI误解)

  身份差带来的 obedience 变化 → 描述为"照做""听从安排""按吩咐办", 禁止"被强迫""被迫屈服"
  感情驱动的 obedience 变化 → 描述为"愿意为他做""不想让他失望""心甘情愿"
  恐惧驱动的 obedience 变化 → 仅此情况允许"被胁迫""不敢反抗"等叙事
  防AI误解: "她对{{user}}的服从度从20升到50" ≠ "她被控制住了"——她可能只是更愿意配合

§21 房产规则

  写入路径: replace /properties 整个容器(合并新旧条目), 计数器: /_uid_counters/property
  property_type: 住宅/商铺/写字楼/厂房/土地
  ownership: 自有(全额购入)/租赁(按月付租)/按揭(按月付贷)
  monthly_payment: 月供或月租金额(自有则为0)
  current_value: 当前市场估值(可随时间/市场变化 update)
  获得房产时创建记录, 出售/退租时删除记录

§22 车辆规则

  写入路径: replace /vehicles 整个容器(合并新旧条目), 计数器: /_uid_counters/vehicle
  vehicle_type: 小汽车/摩托车/货车/自行车
  ownership: 自有/租赁/按揭
  current_value: 当前市场估值
  获得车辆时创建记录, 出售/退租时删除记录

§23 公司规则

  写入路径: replace /companies 整个容器(合并新旧条目), 计数器: /_uid_counters/company
  ownership_pct: {{user}}持股比例(0~100)
    100 = {{user}}全资, 50 = 对半合资, 0 = 已退出但记录保留
  status: 筹备中/正常经营/停业/已注销
  total_value: 公司估值
  公司经营数据(annual_revenue/profit)应随故事发展更新
  控股关系: 若 comp_A 持有 comp_B 股份, 写入 company_holdings 数组

§24 持股规则

  company_holdings: 公司间持股关系
    { from_comp: "comp_NNN", to_comp: "comp_NNN", shares_pct: 0~100 }
  {{user}} → 公司A → 公司B 形成两层持股链
  写入: replace /company_holdings 为完整数组
