§0 输出格式

容器:
  open_tag:  <UpdateVariable>
  inner_tag: <JSONPatch>
  payload:   一个 JSONPatch 数组
  close_tag: </JSONPatch></UpdateVariable>

操作:
  add:     { op: "add",     path: "/...", value: ... }
  replace: { op: "replace", path: "/...", value: ... }
  remove:  { op: "remove",  path: "/..." }
  数组追加: path 以 /- 结尾, value 为单元素

禁止:
  - 用 markdown 围栏包裹 JSON
  - 整对象覆盖(改单字段只 patch 该字段)
  - JSON 注释
  - 跳过UpdateVariable容器——即使本轮无任何变量变化，也必须输出完整容器，JSONPatch数组为空 []。不可省略 <UpdateVariable> 和 <JSONPatch> 标签

§3 类型一致性

数字: 不带引号, 不带千位分隔符, 小数字不用科学计数法
   200   "200"   "1,200"   2e2
布尔: 仅 true / false, 不用 "是"/"否"/"1"/"0"
   true   "是"   "false"(带引号)
字符串: 必须带引号
数组: 用 [] 包裹, 不用顿号串
   ["tag1", "tag2"]   "tag1, tag2"
null: 不带引号
   null   "null"   "无"

§4 数值范围

affection: 0~100, 初始=0
obedience: 0~100, 初始=0
age: >=0, 真实年龄
cycle_length: >=1, 默认28
period_duration: >=1, 默认5
经验计数(total/oral/vaginal/anal): >=0
orgasm_count: >=0
_uid_counters.*: >=0
stats.*: 0~100(physical/mental/fatigue/hunger/hygiene)

§5 Enum 字段

current_phase: 仅限 "月经中" / "安全期" / "受孕期"
   "排卵期" "经期" "危险期" "黄体期" -> 必须用上述三者之一

§6 联动规则

6.1 时间推进联动
  更新 /time/day -> 必须同步更新 /time/weekday
  更新 /time/month -> 需检查是否跨月, 跨月重置 day=1
  更新 /time/year  -> 需检查是否跨年, 跨年重置 month=1
  改任何时间字段 -> 更新所有女性NPC的 menstrual.current_phase

6.2 月经相位计算
  根据 /time 当前日期 与 NPC.last_period_date 的差值计算:
    差值 = 当前日期 - last_period_date(to days)
    phase_day = 差值 % cycle_length
    若 phase_day < period_duration  -> "月经中"
    若 period_duration <= phase_day < period_duration + 6(估算受孕窗口) -> "受孕期"
    否则 -> "安全期"
  时间推进时必须对所有 sexual.*.menstrual 执行此计算

6.3 UID计数器联动
  新建 /sexual/char_NNN -> /_uid_counters/sexual_char +1
  新建 /npc_male/mnpc_NNN -> /_uid_counters/npc_male +1
  新建 /profile/relationships/rel_NNN -> /_uid_counters/relationship +1

6.4 首次性行为联动
  女性NPC首次 vaginal/oral/anal 经验从0变为正数:
    is_virgin -> false(若原本true)
    若 first_partner 为 null -> 设为 {{user}}名字

6.5 计数联动
  发生性行为 -> experience.对应类型+1, experience.total+1
  达到高潮 -> orgasm_count+1

§7 输出前自检

A 路径合法: 每段路径在 §1 schema 树中存在, key 用 UID 格式, 没漏中间层
B 类型范围: 数字无引号/不越界, bool 非"是否", pct 0~100 内
C 联动同步: 改时间->同步星期+月经相位, 新实体->计数器+1
D 不整对象 replace: 改单字段只 patch 目标路径, 不改上级对象
E 剧情-变量同步: 正文出现的会复现的实体, 变量里必须有
F 反流水账: 存的必须是"明天还成立"的状态(见 §9 语义类型表)
G 言灵门控: kotodama 写入前确认 {{user}} 明确以"使用言灵/言灵:"等前缀触发

§8 UID 规范

格式: 前缀_NNN (三位数字序号, 从001开始)
前缀表:
  char_  -> 女性NPC (sexual)
  mnpc_ -> 男性NPC (npc_male)
  rel_  -> 人际关系 (profile.relationships)

闭环:
  1. 按名字扫描全库: 检查是否已有同名人
  2. 命中: update(不新建), 不增计数器
  3. 未命中: 读计数器 -> 计数器+1 -> add 新记录

禁止: key含英文句点 / 用名字做key / 重复UID

§9 建档门槛

不建档(负面清单):
  - 未报真名
  - 功能性过场(收银员/路人/司机)
  - 本轮即离场不再出现
  - {{user}}的父母/亲戚/直系血亲

建档(正面清单):
  - 报出真名 + 有实质互动
  - 明确会再登场
  - {{user}}表现出深入互动意图

男性NPC建档: 一次填满 { name, age, occupation, social_status }
  social_status 描述与{{user}}当前社交关系, 如"室友/火星时代同学/求职同行"

女性NPC建档: 一次填满全部字段(§A全部), 不写占位符/"待定"/空串
  未知字段填基于已有信息的合理推断值, 如:
    年龄 -> 根据外貌/职业推断
    身体部位 -> 根据体型/年龄/着装风格合理描写
    月经 -> 使用默认值(cycle=28, duration=5, phase=安全期)
    性经验 -> 全0, is_virgin=true, first_partner=null(默认)

§10 名人数据参考规则

当故事中出现现实名人时:

10.1 基础数据(必须真实):
  姓名, 出生日期, 年龄: 与公开资料严格一致
  籍贯, 职业, 经历: 以该时间点(当前故事日期)的真实状态为准
  例: 2013年12月出现刘亦菲-> 26岁, 已出演《仙剑奇侠传》《神雕侠侣》
      尚未出演《花木兰》(那是2020年)

10.2 身体数据(尽量真实):
  身高, 三围等: 查阅公开资料, 无法获取时据照片/视频合理推断

10.3 色色数据(合理虚构):
  无法从公开资料获取的私密信息(is_virgin/first_partner/性经验/部位状态):
  据人物公众形象、年龄、公开感情史合理虚构

10.4 处女状态默认规则:
  默认 is_virgin = true, 以下情况除外:
    - 该名人在当前时间线有明确的公开男友/丈夫(含已婚/离婚状态)
    - 有公开报道承认的过往恋爱关系
    - 有子女
  满足除外条件时, is_virgin = false, first_partner 设为已知对象姓名或据公开信息合理推断

10.5 禁止虚构八卦:
  禁止使用搜索引擎查不到的、未经公开报道证实的花边新闻、绯闻、私密传闻
  禁止编造该名人从未被公开报道的恋情或性经历
  personality 字段中的虚构部分必须标注"基于公开形象虚构"

§11 反流水账约束

命中的高危字段:
  personality: 存角色性格的抽象描述, 不存/追加具体行为事例
  appearance/outfit: 存当前外在形象的静态描述, 不含位置/动作/事件
  status(physical/mental): 每轮完全 replace, 只写当前瞬态
  fetishes: 添加时写抽象性癖名(如"被支配"), 不写具体场景事件

判别: 每次写入前自问"这条明天还成立吗？"

§12 清理规则

finance.transactions: 保留最近50条, 超限删除最早的
其余字段: 无滚动窗口上限, 由叙事自然管理

禁止删除:
  - 已被建档的角色条目(除非角色死亡或永久离开舞台)
  - _uid_counters 的任何字段

§13 言灵记录规则

13.1 触发门控（硬门——不满足此条件绝不记录）

  kotodama.records 仅在以下条件**全部满足**时写入:
    A. {{user}}在内心明确以"使用言灵"为意图前缀
       例: "使用言灵——让这杯水变成咖啡"
       例: "言灵: 窗外开始下雪"
       例: "我要用言灵让..."
    B. 该意图指向一个具体的、可验证的现实改变
    C. 改变的生效/失败/部分生效在本轮正文中得到确认

  以下情况**绝不**记录为言灵:
    - 普通愿望/期盼("希望能找到工作""好想吃火锅")
    - 内心os/吐槽/日常念头("这把键盘真难用")
    - 情绪反应("我真想揍他"——除非明确以言灵方式使用)
    - {{user}}的正常行动/决定/选择
    - 对未来的预测或推理(那是先知能力，不是言灵)

13.2 新建记录

  新言灵记录 add 到 /kotodama/records/-:
    id: 基于已有 records 数量递增, 从 1 开始
    time: 当前 /time 时间, 格式 "YYYY-MM-DD HH:MM"
    command: {{user}}言灵的具体内容(原句或精炼表述)
    result: 此生效果描述(如"成功——中行卡余额变为一亿元")
    active: true

13.3 撤销记录

  当{{user}}明确使用言灵撤销/取消之前的某项言灵效果时:
    - 将对应记录的 active 改为 false(replace 操作)
    - 在 result 末尾追加 " | 于 YYYY-MM-DD HH:MM 撤销"
    - 不删除记录, 不改变 id

  当新言灵的效果与旧言灵不可共存时(如先让天晴再让下雨):
    - 旧记录自动 active=false, result 追加 " | 被后续言灵覆盖"

13.4 输出前自检(言灵专项)

  A 触发门控: {{user}}是否明确以"使用言灵"开头?
  B 效果确认: 正文中是否体现了现实改变?
  C 不可共存检查: 新效果是否与现有 active 记录冲突?

§14 数值状态更新规则

14.1 字段范围

  /profile/stats:
    physical: 0~100  # 体力——消耗于体力活动, 恢复于休息/睡眠/进食
    mental:   0~100  # 精神——消耗于压力/高强度思考/社交, 恢复于放松/睡眠/愉悦事件
    fatigue:  0~100  # 疲劳——随时间/活动增长, 睡眠后重置
    hunger:   0~100  # 饱腹——随时间下降, 进食后回升
    hygiene:  0~100  # 卫生——随时间/运动下降, 洗漱/沐浴后回升

14.2 更新时机

  不要求每轮对话都更新。仅在以下情况触发:
    - 场景发生明显的时间推进(跨小时)
    - 发生了消耗/恢复性的行动(运动/进食/睡眠/洗澡/高强度工作)
    - 状态变化幅度>=5 时才 patch

14.3 典型变化量参考

  体力:
    轻微活动(走路/聊天): -0~5
    中等劳动(搬东西/长时间站立): -5~15
    高强度运动(跑步/重体力): -10~30
    睡眠/充分休息: +20~40
  精神:
    轻微用脑(日常对话): -0~3
    高专注(建模/学习/面试): -5~15
    情绪冲击(好消息/坏消息/冲突): ±5~20
    放松/听音乐/散步: +5~10
    睡眠: +15~30
  疲劳:
    每经过约 1 小时清醒: +3~8
    体力消耗大时额外: +5~10
    睡眠: 重置为 5~15
    短暂休息/小憩: -10~20
  饱腹:
    每经过约 2~3 小时: -5~10
    正常一餐: +25~40
    零食: +5~10
  卫生:
    每经过约 4~6 小时: -3~5
    运动/出汗后额外: -5~10
    洗漱/淋浴: +30~50
    换衣服: +10~15

14.4 禁止

  禁止超出 0~100 范围
  禁止不根据剧情凭空调整数值
  禁止在无实际活动变化的对话轮次中机械更新所有五项

§15 位置追踪规则

15.1 字段

  /location: string  当前精确位置

15.2 更新时机

  {{user}}移动到新地点时立即 replace:
    - 进入建筑物: "海淀区学院路XX银行ATM自助区"
    - 进入店铺: "王府井百货大楼3层男装区"
    - 回到住所: "火星时代校外宿舍楼3号楼502室"
    - 在交通工具上: "学院路上一辆伊兰特出租车内"

15.3 精度要求

  必须包含可辨识的地标/建筑名/房间号, 不能只写城市名
  正确: "朝阳区金融街丽思卡尔顿酒店大堂"
  错误: "北京" / "酒店" / "外面"
  如果不知道精确地址, 用可见参照物: "学院路某银行ATM隔间"
